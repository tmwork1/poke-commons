作成日: 2026-07-23
改訂日: 2026-07-23(**v2**。プロダクトオーナーからの方針転換を受けて全面改訂。旧版は「ローカル(IndexedDB)完結」方針だったが、v2では「ログイン + サーバー(Supabase)管理」方針に転換する。差分の要点は本文中に随時記載)

## 1. ゴールとスコープ

初期構想・開発プランでは「育成ビルダー」(`builds` テーブル)を用意しているが、これは**共有用の使い捨てスナップショット**であり、レベル・ニックネーム・複数匹といった「実際に育成した個体を継続して管理する」概念を持たない。battle-assistant-sv の `manager.html` が担っていた役割 — ユーザーが自分の手持ちポケモンを登録・編集・一覧管理する機能 — が poke-commons には存在しない。本計画はこのギャップを埋める。

v2ではこれに加えて、battle-assistant-sv の「ダメージ計算ウィジェット」相当の機能 — **1個体に対して複数の対戦相手ごとの想定ダメージ計算(相手の型・フィールド条件・使用技・計算結果)を紐付けて保存する**機能をスコープに含める。この対戦相手メモは、保存のたびに匿名化コピーを既存の集合知DBパイプライン(`damage_calcs`/`events`)へ還元する。

### 作るもの

- **Supabase Auth によるログイン**、およびログイン後のみ利用できる「個体管理」画面(自分の育成済みポケモンの登録・一覧・編集・削除)
- 個体データは **Supabase(Postgres)側でユーザーに紐づけて永続化**する(§2・§3)
- **個体ごとの対戦相手メモ**: 1個体に対して複数の対戦相手の想定ダメージ計算設定(相手の型・フィールド条件・使用技・計算結果)を紐付けて記録・編集・削除できるUI。既存の `/damage-calc` の計算エンジン(Pyodide)をそのまま呼び出す
- 対戦相手メモの**保存・更新のたびに個人情報を除去した匿名コピー**を `damage_calcs`/`events` へ二重記録する匿名化パイプライン(§4)
- 個体データを既存の `builds`(共有用)へ変換して公開共有する導線、および `builds` を個体として取り込む導線(§5、双方サーバー側で完結)
- (任意機能・格下げ)可搬性のための軽量なJSONエクスポート(§7)

### 作らないもの(非スコープ)

- **OCR/画像解析による個体情報の自動読み取り**(§9で詳述。将来検討事項に留める)
- **battle-assistant-sv 形式のバックアップからの移行インポート**(§9。旧アプリは古く対応不要というプロダクトオーナー判断)
- パーティ編成・ダメージ計算のリアルタイムシミュレーション機能そのもの(既存 `/damage-calc` との連携は行うが、計算エンジンの新規開発は本計画に含まない)
- **ログインスコープの拡大**: ダメージ計算機・検索・育成ビルダー(`builds`)・匿名イベントログ(`events`/`searches`)は本計画の対象外であり、v2でも**引き続き完全匿名のまま変更しない**(§2.1)
- ソーシャル機能(他ユーザーの個体を閲覧する・フォローする等)。個体管理は本人限定のプライベートな機能とする

## 2. アーキテクチャ方針

### 2.1 認証 — Supabase Auth を「個体管理」機能のみにスコープ限定して導入

#### 方針転換の経緯

旧版(v1)は「poke-commons には認証機構が一切ない」ことを前提に、個体データをブラウザ(IndexedDB)へ置く方針(旧§2.1の案A)を採用していた。プロダクトオーナーからの方針転換により、v2では**個体データの永続化をサーバー(Supabase/Postgres)側で行い、ユーザーはログインして自分の個体を管理する**方式に変更する。

#### 認証方式の選定

| 案 | 内容 | 評価 |
|---|---|---|
| **Google OAuth(採用)** | Supabase Auth の Google プロバイダを利用したソーシャルログイン | パスワード管理・パスワードリセット運用が不要。多くのユーザーが既にGoogleアカウントを保有しており、個体管理機能を使いたいユーザーにとって最も摩擦が少ない。導入コストはGoogle Cloud Console側でのOAuthクライアント登録・リダイレクトURI設定が必要になる分、メールマジックリンクよりやや高いが、Supabase Auth側の設定自体は標準サポートされており大きな負担ではない |
| メールマジックリンク | Supabase Auth の Magic Link / OTP を利用。パスワード管理不要 | `@supabase/supabase-js` のみで完結し外部IdP登録が不要な分、導入コストは最小。ただしメール到達性(迷惑メール振り分け等)に体験が左右される |
| パスワード認証 | メール+パスワードのSupabase Auth標準機能 | 実装は簡単だが、パスワードリセット・強度要件・漏洩対策など運用負荷が増える。個体管理という限定スコープに対して過剰 |

**結論: Supabase Auth の Google OAuth を採用する(プロダクトオーナー指定、2026-07-23)。** Google Cloud Console側でのOAuthクライアントID発行・承認済みリダイレクトURIの設定(本番/開発それぞれ)が事前準備として必要になる。将来的にGoogleアカウントを持たないユーザーからの要望が出た場合は、メールマジックリンクを追加のログイン手段として併設することを妨げない(必須要件ではない)。

#### スコープ限定という設計判断とそのトレードオフ

**ログインは「個体管理」機能のみのスコープとする。** 既存のダメージ計算機・検索・育成ビルダー(`builds`)・匿名イベントログ(`events`/`searches`/`damage_calcs`)は引き続き完全匿名のまま変更しない。

この判断の論拠は、初期構想([初期構想.md](./初期構想.md))が掲げる「匿名運用が核」という既存思想との衝突を最小化することにある。個体管理以外の全機能が匿名のまま維持されれば、poke-commonsを訪れる大多数のユーザー(ダメージ計算・検索・共有ビルド閲覧が目的の層)にとっては何も変わらず、認証の存在自体が意識されない。ログインを要求されるのは「自分の手持ちポケモンを継続管理したい」という能動的な意思を持つユーザーのみである。

ただし、この判断は**思想的にトレードオフを伴う**ことを明記する。

- `migrations/002_enable_rls.sql` の冒頭コメントは「poke-commons はログイン無しの完全匿名サービスのため、`auth.uid()` ベースの『本人限定』ポリシーは存在しない」と明言している。v2はこの前提を破り、`owned_pokemon`/`opponent_notes` の2テーブルに関して**初めて認証済みユーザー本人限定のRLSポリシー**を導入することになる(§3.3)。
- 「個体管理という一機能のためだけに認証基盤を新設する」ことは、匿名運用を核とする既存プロダクトの一貫性に対する例外を作る判断であり、軽視すべきではない。
- 上記を踏まえ、**本計画のうち認証基盤の新設(§2.1・§3・§4)は、正式な開発プランへの統合(§11)の前にプロダクトオーナーの最終承認を要する事項として明記する。** Phase A着手前に改めて承認を得ること。

### 2.2 集合知思想との整合性

初期構想の3原則との関係は以下のとおり整理できる。

| 原則 | 育成個体管理での扱い |
|---|---|
| Every Calculation Matters | 対戦相手メモとして記録される想定ダメージ計算は、匿名化パイプライン(§4)を通じて既存の `damage_calcs`/`events` へそのまま還元される。個体そのもの(ニックネーム・タグ等の実在属性)は非公開の本人データとして扱う |
| Collective Intelligence | 個体管理機能自体はユーザー個人の生産性ツールだが、そこから生まれる対戦相手メモの計算条件は自動的に集合知へ還元される。旧版(v1)の「共有導線を通じてのみ貢献」から一歩進み、**利用するだけで貢献が発生する**設計になる |
| Data First | 蓄積すべきは行動ログ(=計算条件)であり個人の所有物ではない、という原則は維持する。`owned_pokemon`/`opponent_notes` はユーザーの所有物としてサーバーに保存するが、これは`events`等の集合知テーブルとは明確に分離された非公開データであり、集合知への還元は§4の変換関数を経由したコピーのみで行う |

### 2.3 既存機能との関係を壊さない

- `builds`/`damage_calcs`/`events`/`searches`/`suggestions` の既存テーブル定義・`session_hash` の生成ロジック・既存のRLSポリシー(`anon`/`authenticated` ロールへの公開SELECTのみ、書き込みは `service_role` 経由のAPIのみ)には**変更を加えない**
- `owned_pokemon`/`opponent_notes` への書き込みAPI自体も、既存の `/api/damage-calcs`・`/api/builds` と同様に Astro のサーバーサイドAPIルートとして実装し、**Supabase呼び出しは `getSupabaseAdminClient()`(service_role)で行う**。これにより「書き込みは全てservice_role経由のサーバーAPIのみ」という既存の設計原則(`migrations/002_enable_rls.sql`)は崩さない。`owned_pokemon`/`opponent_notes` のRLSが `auth.uid()` を要求するのは、あくまで**ユーザー本人のブラウザから直接読み書きする経路を想定しない**(APIを必ず経由させる)場合でも、将来クライアントSDKから直接アクセスする経路を開ける設計上の保険として持たせる、という位置づけである
- 個体 → build 変換(§5)は既存の `POST /api/builds` を呼び出す形に留め、`builds` テーブル自体・そのAPIの実装には変更を加えない

## 3. データモデル(Postgres)

v1で検討していた「育成個体のブラウザ内(IndexedDB)保存」は撤回し、v2では以下2テーブルを `migrations/NNN_*.sql` として新設し、**Postgres側**にユーザー紐づきで永続化する。

### 3.1 `owned_pokemon`(仮称・所有個体)

battle-assistant-sv の `Pokemon` クラス(`js/pokemon.js`)を土台にしたフィールド構成はv1から概ね踏襲しつつ、保存先をPostgresのテーブル定義として書き直す。

| カラム | 型 | 説明 |
|---|---|---|
| `id` | `uuid` | 主キー(`gen_random_uuid()`) |
| `user_id` | `uuid` | `references auth.users(id) on delete cascade`。所有者 |
| `nickname` | `text` | 表示名。空なら図鑑名を表示に流用 |
| `species_name` | `text not null` | 図鑑名(既存マスタ `public/master-data/` を参照) |
| `level` | `int` | レベル(1〜100) |
| `nature` | `text` | 性格 |
| `ability_name` | `text` | 特性 |
| `item_name` | `text` | 持ち物 |
| `tera_type` | `text` | テラスタイプ |
| `evs` | `jsonb not null default '{}'` | 努力値(既存 `builds.evs` と同形式。チャンピオンズ形式・0〜32刻み) |
| `ivs` | `jsonb not null default '{}'` | 個体値。既定31 |
| `move_names` | `text[] not null default '{}'` | 技構成(最大4、既存 `builds.move_names` と同形式) |
| `memo` | `text` | 自由記述メモ |
| `tags` | `text[] not null default '{}'` | フラットなタグ(多対多、階層フォルダにしない) |
| `is_pinned` | `boolean not null default false` | ピン留め/お気に入り |
| `source_build_slug` | `text` | この個体が `builds` から取り込まれた場合の元 `share_slug`(履歴参照用、任意) |
| `created_at` | `timestamptz not null default now()` | 作成日時 |
| `updated_at` | `timestamptz not null default now()` | 最終更新日時(並び替えに使用) |
| `last_used_at` | `timestamptz` | 最終使用日時(対戦相手メモ作成・ダメージ計算機読み込み時に更新、並び替えに使用) |

**非対応として明記するフィールド**: 色違い・リボン・捕獲情報・図鑑ID等のコレクション要素、`sex`/`weight`(battle-assistant-svで形骸化していた未使用フィールド)は poke-commons でも対象外とする。

### 3.2 `opponent_notes`(仮称・対戦相手メモ)

battle-assistant-sv の「ダメージ計算ウィジェット」相当の機能を、個体に紐づく複数レコードとしてモデル化する。

| カラム | 型 | 説明 |
|---|---|---|
| `id` | `uuid` | 主キー(`gen_random_uuid()`) |
| `owned_pokemon_id` | `uuid` | `references owned_pokemon(id) on delete cascade`。紐づく所有個体 |
| `user_id` | `uuid` | `references auth.users(id) on delete cascade`。RLS簡素化のための非正規化(`owned_pokemon`経由のJOINに依存しない） |
| `opponent_build` | `jsonb not null default '{}'` | 相手の想定構成(種族・性格・特性・持ち物・テラス・努力値・個体値・技など) |
| `field` | `jsonb not null default '{}'` | 天候・地形・壁等のフィールド条件 |
| `move_name` | `text` | 自分側(`owned_pokemon`)が使用する技 |
| `client_result` | `jsonb` | Pyodide計算結果のスナップショット(UI表示用の未検証値、既存 `damage_calcs.client_result` と同じ扱い) |
| `memo` | `text` | 自由記述メモ |
| `created_at` | `timestamptz not null default now()` | 作成日時 |
| `updated_at` | `timestamptz not null default now()` | 最終更新日時 |

### 3.3 RLSポリシー方針

`owned_pokemon`・`opponent_notes` は、既存テーブルとは異なり**閲覧も含めて完全に非公開**とする。

- `SELECT`/`INSERT`/`UPDATE`/`DELETE` の全操作を `auth.uid() = user_id` の本人限定ポリシーとする(`anon`ロールへの公開SELECTポリシーは一切作らない)
- 既存の `002_enable_rls.sql` にある「poke-commonsには`auth.uid()`ベースの本人限定ポリシーは存在しない」というコメントを、v2適用時にあわせて更新する必要がある(実装時のタスクとして明記。§8 Phase B)
- Phase B完了条件として、**本人以外のユーザーが他人の `owned_pokemon`/`opponent_notes` を閲覧・編集できないことを確認するRLSポリシーテストを必須化**する(§10リスク表)

## 4. 匿名化パイプライン — 対戦相手メモから既存集合知DBへの二重記録

これがv2の中核的な新規要素である。対戦相手メモ(`opponent_notes`)の保存・更新のたびに、個人情報を除去した匿名コピーを既存の `damage_calcs`/`events` へ書き込み、集合知DBへ還元する。

### 4.1 全体の流れ

```
[owned_pokemon(本人限定)]                [opponent_notes(本人限定)]
  species_name, nature, ability_name,      opponent_build, field,
  item_name, tera_type, evs, ivs,          move_name, client_result
  move_names  ─┐                        ┌─
               │  ホワイトリスト変換関数  │
               ▼                        ▼
        ┌─────────────────────────────┐
        │  attacker_build / defender_build │  ※自分側=attacker想定。
        │  field / move_name / client_result│    実装時に攻守どちらに
        └─────────────────────────────┘    立つ想定かは技整合性で決定
               │  user_id・nickname・tags・is_pinned等は一切含めない
               ▼
[damage_calcs(既存・匿名)] + [events(既存・匿名)]
   session_hash: 実装時に決定(§4.3)
```

### 4.2 API設計

`POST /api/opponent-notes`(新規作成)・`PUT /api/opponent-notes/:id`(更新)等のエンドポイントで、以下を行う。

1. 認証済みユーザーのリクエストを検証(`auth.uid()`)
2. `opponent_notes` レコードを保存(本人限定テーブルへの通常のCRUD)
3. 上記1と**同一トランザクション内、または直後の非同期処理として**、匿名化コピーを `damage_calcs` + `events` へ二重記録する

3の実装は、既存の `POST /api/damage-calcs`(`src/pages/api/damage-calcs.ts`)・`POST /api/builds` が既に採用している「**主目的の書き込みが成功していれば、副次的な集計用の書き込みが失敗してもリクエスト全体は失敗させない**」パターンをそのまま踏襲する。`opponent_notes` の保存が主目的、`damage_calcs`/`events` への匿名記録が副次記録という位置づけである。

### 4.3 変換関数(ホワイトリスト方式) — 実装上の必須要件

匿名化コピーを作る変換関数は、**渡すフィールドを明示的にホワイトリスト列挙し、それ以外は絶対に渡さない**方式で実装する。ブラックリスト方式(「これを除外する」列挙)は将来のフィールド追加時に個人情報が漏れるリスクがあるため採用しない。

- `owned_pokemon` から渡してよいフィールド: `species_name`・`nature`・`ability_name`・`item_name`・`tera_type`・`evs`・`ivs`・`move_names` のみ
- `owned_pokemon` から**渡してはいけない**フィールド(明記): `id`・`user_id`・`nickname`・`tags`・`is_pinned`・`memo`・`source_build_slug`・`created_at`・`updated_at`・`last_used_at`
- `opponent_notes` から渡してよいフィールド: `opponent_build`・`field`・`move_name`・`client_result` のみ
- `opponent_notes` から**渡してはいけない**フィールド(明記): `id`・`owned_pokemon_id`・`user_id`・`memo`・`created_at`・`updated_at`

この変換関数は単一箇所(例: `src/lib/opponent-note-anonymize.ts`)に実装し、Phase Dでの単体テストにより「ホワイトリスト外のフィールドを混入させようとしても出力に現れない」ことを担保する(§10リスク表)。

### 4.4 `session_hash` の扱い(実装時に決定)

`damage_calcs`/`events` の `session_hash` 列は非NULL制約があるが、`owned_pokemon`/`opponent_notes` はログインユーザーの操作であり、既存の匿名セッションCookie(`pc_sid`)由来の `session_hash` 発行ロジック(`src/lib/session-hash.ts`)とは文脈が異なる。以下のいずれかを実装時に既存コードとの整合性を見て決定する。

- 既存のセッション発行ロジック(`computeSessionHash`)をログインユーザーにもそのまま適用し、匿名セッションCookieを流用する
- ログイン由来の記録であることを示す固定マーカー文字列、またはユーザーと紐付かない別ハッシュをNOT NULL制約回避のために用いる

いずれにせよ **`user_id` そのもの、または `user_id` から一意に逆算可能な値を `session_hash` に用いてはならない**(匿名化の趣旨に反するため)。

## 5. builds(既存)との関係

### 5.1 設計方針: 統合せず、一方向の射影で連携する(v1から概念は維持)

| 観点 | `builds`(既存・共有用) | 育成個体(`owned_pokemon`・本計画) |
|---|---|---|
| ライフサイクル | 書き捨ての不変スナップショット | 長期にわたり編集され続ける可変レコード |
| 所有権 | 匿名(`session_hash`、user_idなし) | ログインユーザー本人所有(`user_id`) |
| 保存場所 | サーバー(Supabase、`builds`テーブル) | サーバー(Supabase、`owned_pokemon`テーブル) |
| 主キー | `id`(uuid)、公開URLは`share_slug` | `id`(uuid、`owned_pokemon`独自) |
| レベル・ニックネーム・タグ・ピン留め | 持たない | 持つ |

v1では個体側がブラウザ(IndexedDB)にあったため「ローカル→サーバー」の変換だったが、**v2では双方がPostgres上にあるため、IndexedDBを介さずAPI間の変換で完結する**。ライフサイクル・所有権(匿名 vs 本人)のいずれも異なるため、テーブルの統合は行わず、v1と同じく**個体(instance)→設計図(build)への一方向の射影**として連携する。

```
[owned_pokemon(Supabase、本人所有・非公開)]
   nickname, level, tags, memo, is_pinned, source_build_slug ...
        │
        │  「共有する」操作(明示的なユーザー操作、実在属性を落とす)
        ▼
[builds(Supabase、匿名共有)]
   pokemon_name, nature, ability_name, item_name, tera_type, evs, ivs, move_names, share_slug
```

### 5.2 連携導線

- **build → 個体への取り込み**: 既存の育成ビルダー(`/builds/new`)で作成した育成、または他ユーザーが共有した `share_slug` の育成を、ログイン済みユーザーが「個体として取り込む」ボタンで `owned_pokemon` へコピーする(サーバー側APIで `builds` 行を読み取り `owned_pokemon` へINSERT)。取り込み時に `source_build_slug` を記録し、ニックネーム・レベル・タグをユーザーに追加入力させる
- **個体 → build への共有**: 個体一覧・編集画面から「共有する」を選ぶと、既存の `POST /api/builds` を呼び出して匿名化済みの `builds` レコードを新規作成し、`share_slug` を発行する。既存APIの変更は不要(呼び出し元が増えるのみ)
- 双方向とも**個体側から明示的に操作した場合のみ**発生する片方向の変換であり、`builds` 側の自動同期・逆反映は行わない(builds は不変スナップショットのまま)

## 6. UI/UX要件

### 6.1 一覧表示(v1から維持)

- **並び替え**: 最終更新順(`updated_at`)・最終使用順(`last_used_at`)・ニックネーム順を切り替え可能
- **タグフィルタ**: フラットなタグの複数選択AND/OR切り替え(階層フォルダ化はしない)
- **ピン留め/お気に入り**: `is_pinned` 個体を一覧上部に固定表示
- 文字列検索: ニックネーム/種族/タイプ/特性/持ち物/テラス/技のAND部分一致

### 6.2 編集フォーム — 即時自動保存(保存先の変更に伴う調整)

battle-assistant-svの「全項目が変更のたびに即座に自動保存され、明示保存ボタンがない」UXは踏襲する価値があると判断し、v1同様に維持する。ただしv2では保存先がIndexedDBの同期書き込みからAPI経由の非同期書き込みに変わるため、以下を追加で考慮する。

- **デバウンス**: 入力のたびにAPIを叩くとレートリミット・DB負荷の両面で非効率なため、既存のダメージ計算UI等で採用している程度のデバウンス(数百ms〜1秒程度)を挟んでからPATCHする
- **楽観的UI更新**: 入力値は即座に画面へ反映し、API呼び出しはバックグラウンドで行う。保存失敗時のみエラー表示とリトライ導線を出す(保存中であることをブロッキングさせない)
- 誤操作からの復帰手段(直前状態への「元に戻す」)は、v1同様Phase内での実装余地として扱う(必須要件ではない)
- 種族・技・特性・持ち物の入力補助は独自コンボボックスを新規実装せず、既存の検索/詳細ページ(§6.3)のマスタデータ・オートコンプリート資産を再利用する

### 6.3 既存の検索・詳細ページとの連携(v1から維持)

- 種族名入力時のオートコンプリートは `public/master-data/` の軽量JSON(既存の検索API `POST /api/search` が使っているものと同一)を参照し、マスタデータを二重管理しない
- 個体編集画面から該当ポケモンの `/pokemon/[name]` へのリンクを表示し、種族値確認の導線を作る。技選択も同様に `/moves/[name]` への参照導線を設ける

### 6.4 対戦相手メモUI(v2新規)

- 個体詳細/編集画面から、その個体に紐づく対戦相手メモの一覧・追加・編集・削除ができる
- メモ追加時は、既存の `/damage-calc` の計算エンジン(Pyodide)をそのまま呼び出し、相手の型(`opponent_build`)・フィールド条件(`field`)・使用技(`move_name`)を入力すると計算結果(`client_result`)が算出される導線とする(計算エンジンの新規実装は行わない、§1非スコープ)
- 保存・更新のたびに§4の匿名化パイプラインが動作することはユーザーには明示しない(既存の `/damage-calc` の計算ログ蓄積と同じ扱いとし、通常の計算機利用時と同様に静かに行う)。ただしプライバシーポリシー等の文言更新が必要かは別途検討事項とする

## 7. インポート/エクスポート(任意機能に格下げ)

v1では「ローカル完結」方針の必須条件としてエクスポート/インポートを一級機能と位置づけていたが、v2ではサーバー管理(Supabase永続化)になったことで**ローカル消失リスクを埋めるための必須機能という位置づけは不要になる**。

- **任意機能**として、個体データを他サービスへ引っ越す際の可搬性、またはユーザー自身がデータを確認するためのダウンロード用途に限り、軽量なJSONエクスポートを残してよい(バージョン番号付きJSON形式、`owned_pokemon`/`opponent_notes` を含める)
- インポート(他形式からの取り込み)は本計画の必須要件としない。**battle-assistant-sv形式からの移行インポートは対応除外**(§9)
- 優先度は低く、Phase A〜Eの完了後、余力があれば着手する程度の扱いとする(§8)

## 9. 非スコープの明記

### 9.1 OCR自動登録

battle-assistant-svの目玉機能である「Switchキャプチャ映像からのOCR/テンプレートマッチング(OpenCV.js + Tesseract.js)による個体情報自動読み取り」は、**本計画のスコープに含めない**。

- Switch UIの変更・言語設定・キャプチャ品質に依存する保守コストが継続的に発生し、集合知というpoke-commonsのコア価値に直接寄与しない
- 将来、個体登録数が十分に増えて手入力の負荷が明確なボトルネックになった場合に、改めて別プランとして検討する

### 9.2 battle-assistant-sv形式からの移行インポート

v1(旧§6.3)ではbattle-assistant-svのバックアップ形式(`pokemonStorage`/`appStorage`のJSON)からの移行インポートを費用対効果の高い施策として計画していたが、**v2ではプロダクトオーナー判断により対応除外とする**。

- 理由: battle-assistant-svは旧アプリであり、互換対応のための実装・保守コストに見合う価値がないとプロダクトオーナーが判断した
- 既存のbattle-assistant-svユーザーは、新規に poke-commons 上で個体を登録し直す運用となる。移行支援は行わない
- OCR非スコープ(§9.1)とセットで検討していた「OCR登録→書き出し→取り込み」という移行導線も、本判断により不要になった

## 10. リスクと対応

| リスク | 影響 | 対応 |
|---|---|---|
| 認証基盤導入が「匿名運用」という既存プロダクト思想と部分的に衝突する | プロダクトの一貫性・思想的整合性への疑義 | ログインを個体管理機能のみにスコープ限定(§2.1)。ダメージ計算機・検索・育成ビルダー・匿名イベントログは無変更。**正式な開発プランへの統合前にプロダクトオーナーの最終承認を要する事項として明記**(§11) |
| 対戦相手メモの匿名化漏れ(`user_id`/`nickname`等が`damage_calcs`/`events`に混入するプライバシーリスク) | 個人(のニックネーム等)を特定しうる情報が匿名集合知DBに漏洩し、信頼毀損に直結する | ホワイトリスト方式の単一変換関数(§4.3)で実装し、渡してよいフィールドのみを明示列挙。Phase D-4で「ホワイトリスト外フィールドが出力に含まれない」ことを検証する単体テストを必須化 |
| ログイン導入によるユーザー離脱・フリクション | 個体管理機能の利用率低下 | 低摩擦な認証方式(Google OAuth、パスワード管理不要)を選定(§2.1)。個体管理を使わない一般ユーザー(ダメージ計算・検索・共有ビルド閲覧が目的の層)には認証UIを一切露出させない設計とする(Phase A-2) |
| Googleアカウントを持たない/使いたくないユーザーが個体管理機能を利用できない | 個体管理機能の利用者層がGoogleアカウント保有者に限定される | 初期スコープでは許容(§2.1)。要望が出た場合はメールマジックリンク等の追加ログイン手段を後から併設できる設計とする |
| 新規RLSポリシーの設計ミス(本人以外が閲覧・編集できてしまう等) | 個体データ・対戦相手メモの情報漏洩 | Phase B-3でのRLSポリシーテストを必須完了条件とする(本人以外のアクセスを拒否することをCIで継続的に検証) |
| `opponent_notes`の増加に伴うDB容量・クエリ性能 | 個体詳細画面の表示遅延、DBコスト増 | Phase B-4でのインデックス設計。集計・傾向分析が必要になった場合は開発プラン.md Phase 5の既存バッチ方針(GitHub Actionsによる定期集計)に合流させ、リアルタイム集計はサーバーAPIに持たせない |
| `damage_calcs`/`events`への二重記録の失敗が個体管理の主機能を妨げる | 対戦相手メモ保存自体が失敗しユーザー体験が悪化する | 既存 `damage-calcs.ts`/`builds.ts` と同じ「主目的成功後の副次記録、失敗してもリクエスト全体は失敗させない」パターンを踏襲(§4.2)。副次記録の失敗はログに残しつつユーザーへは主目的の成功のみを返す |
| `session_hash`の扱いを誤り、ログインユーザーを匿名DB上で追跡可能にしてしまう | 匿名運用の前提を実質的に破る | `user_id`そのもの、または`user_id`から一意に逆算可能な値を`session_hash`に用いないことを実装ルールとして明記(§4.4)。Phase D-4のテスト対象に含める |

## 11. 既存開発プランへの統合方法

本計画は認証基盤の新設という**思想レベルの決定**を伴うため、v1同様に独立ドキュメントとして運用を継続する。

- **`docs/開発プラン.md` への正式統合、および `docs/初期構想.md` への匿名運用ポリシーの例外規定(個体管理機能に限りログインを要求する旨)の追記は、プロダクトオーナーの最終承認後に行う。** 承認前にこれら2文書へ手を加えない
- 承認後の統合イメージ: 開発プラン.mdの「進捗サマリー」表に本計画のPhase A〜Eを新規Phaseとして追記し、`§3 開発フェーズ`から本ドキュメントへリンクする形に統合する
- 初期構想.mdへの追記イメージ: 「匿名運用が核」の原則に対する例外として、個体管理機能(ログイン必須)の存在とそのスコープ限定方針(§2.1)を明記する
- 着手判断(Phase A着手可否を含む)がされるまでは、本ドキュメントを唯一の一次情報源として更新し、開発プラン.md・初期構想.md側は変更しない

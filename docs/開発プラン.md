# Poke-Commons 開発プラン

作成日: 2026-07-23
前提資料: [初期構想.md](./初期構想.md)
参考実装: `../poke-research/`(Astro + Cloudflare Workers + Supabase の稼働実績あり)
計算エンジン: `../jpoke/`(純Python・依存ゼロのポケモンチャンピオンズ対戦ライブラリ)

---

## 1. ゴールとスコープ

初期構想の3機能 + データ蓄積基盤を、以下の順で段階的にリリースする。

1. **ダメージ計算機**(+ 計算ログの匿名蓄積)
2. **育成ビルダー**(保存・共有 + 蓄積)
3. **検索**(+ 検索ログ蓄積)
4. **データ活用**(蓄積データからのサジェスト表示)

対象ゲームは jpoke の対応範囲に合わせ **ポケモンチャンピオンズのシングルバトル**(努力値は 0〜32 のチャンピオンズ形式)。

「Data First」の思想に基づき、**イベントロギング基盤を Phase 1 で最初に作り**、全機能はその上に載せる。

## 2. アーキテクチャ

### 2.1 全体構成

```
[ブラウザ]
   │  Astro SSR ページ + フォーム/検索 UI
   ▼
[Cloudflare Workers: メインアプリ (TypeScript / Astro)]
   │  ├─ ファイルベース API (src/pages/api/**)
   │  ├─ middleware.ts (認証・レートリミット・CSRF)
   │  └─ worker.ts (fetch + scheduled 合成エントリ)
   │
   ├──▶ [Cloudflare Workers: 計算サービス (Python Workers + jpoke)]
   │       Service Binding 経由。ダメージ計算/致死率計算 API
   │
   └──▶ [Supabase (Postgres)]
           マスタ参照キャッシュ・イベントログ・育成データ・集計テーブル
```

- **メインアプリ**: poke-research と同じ Astro + `@astrojs/cloudflare` 構成。単一 npm パッケージ(モノレポにしない)。
- **計算サービス**: jpoke を Cloudflare **Python Workers** に載せた別 Worker。メインアプリから Service Binding で呼ぶ(外部公開しない)。
- **DB**: Supabase (Postgres)。ローカル開発は Supabase CLI (`supabase start`, Docker) を使用(初期構想の指定どおり。poke-research には無い部分なので新規セットアップ)。

### 2.2 jpoke 実行方式 — Phase 0 で検証して確定

jpoke はランタイム依存ゼロ・C拡張なしの純 Python のため Pyodide 上で動く見込みが高いが、未実証のリスクがあるため **Phase 0 の技術検証で以下を確定させる**。

| 案 | 内容 | 懸念 |
|---|---|---|
| **A (推奨)** | Python Workers で jpoke を動かす専用 Worker | コールドスタート、同梱 JSON 約3MB のバンドルサイズ制限、`importlib.resources` の互換性、CPU時間制限 |
| B (代替1) | ブラウザ内 Pyodide で jpoke をクライアント実行 | 初回ロードが重い(数十MB)。ただしサーバ負荷ゼロ・オフライン計算可 |
| C (代替2) | ダメージ計算コア部分のみ TypeScript へ移植 | 移植工数大・jpoke の 6174 テストとの等価性維持が困難。最終手段 |

判断基準: 案Aで「コールドスタート含む p95 応答 < 1.5s、バンドルがサイズ制限内、計算結果が jpoke ネイティブと完全一致」なら A 採用。だめなら B → C の順で検討。

### 2.3 マスターデータの流れ

jpoke の `src/jpoke/data/`(pokedex.json / moves.json / learnsets.json / CSV)を**単一の信頼できる情報源**とし、ビルド時スクリプトでフロントエンド用 JSON(ポケモン名・技名・特性・アイテムのオートコンプリート用軽量データ)を生成して静的アセット配信する。DB にはマスタを複製せず、**イベントログには名前(jpoke の日本語キー)をそのまま記録**する。

### 2.4 データモデル(初期案 — Phase 1 で確定)

```sql
-- 匿名イベントログ (append-only、全機能共通の蓄積基盤)
events (
  id bigint generated always as identity,
  event_type text,          -- 'damage_calc' | 'build_save' | 'search' | 'compare' | ...
  payload jsonb,            -- イベント固有データ (計算条件、検索語 など)
  session_hash text,        -- 匿名セッション識別子 (日次ローテーションのハッシュ、個人特定不可)
  created_at timestamptz default now()
)

-- 型付きテーブル (集計しやすい主要イベントは正規化して二重記録)
damage_calcs ( attacker_name, defender_name, move_name, attacker_build jsonb,
               defender_build jsonb, field jsonb, result jsonb, session_hash, created_at )

builds ( id uuid, pokemon_name, nature, ability_name, item_name, tera_type,
         evs jsonb, ivs jsonb, move_names text[], share_slug text unique,
         is_public boolean, session_hash, created_at )

searches ( query text, category text, hit_count int, session_hash, created_at )

-- 集計結果 (cron ジョブが書き込み、閲覧 API は読むだけ)
suggestions ( kind text, subject_key text, payload jsonb, computed_at )
  -- kind 例: 'popular_build' | 'common_opponent' | 'ev_trend' | 'trending_search'
```

- **RLS を Phase 1 の最初から全テーブルに適用**(poke-research では後付け対応 `027_enable_rls.sql` になった教訓)。閲覧系は `SELECT` のみ public、書き込みは service_role 経由の API のみ。
- 匿名化ポリシー: IP・UAは保存しない。`session_hash` は日替わりソルト付きハッシュで長期追跡不可にする。

### 2.5 poke-research から流用するパターン

| 項目 | 流用内容 |
|---|---|
| マイグレーション | `migrations/NNN_*.sql` 連番 + `scripts/db/run-migrations.mjs`(`pg` 直結の自前ランナー、適用済み管理テーブル) |
| API 設計 | Astro ファイルベースエンドポイント + `api/_shared.ts` 共通ヘルパー。Hono 等は導入しない |
| 認証 | 管理系 API のみ Basic 認証(`middleware.ts`)。一般ユーザーはログイン不要(匿名) |
| Cron | `worker.ts` に fetch + scheduled 合成エントリ。**Cron Trigger の上限は Free プランでアカウントごと5件**([公式 Limits](https://developers.cloudflare.com/workers/platform/limits/))で、poke-research が3件使用済み。**残り枠内で対処する方針(決定済み 2026-07-23)**: poke-commons は1件に集約し分スロットでジョブ分岐(poke-research の `DAILY_SLOT_JOBS` パターン) |
| テスト | `node --test`(外部フレームワーク無し)+ CI で postgres:16 コンテナへのマイグレーション DDL 検証 |
| デプロイ | Cloudflare Git 連携で main ブランチ自動デプロイ。本番マイグレーションのみ `npm run release` で手動先行適用 |
| 環境変数 | `.env.example` / `.env` / `.env.production` / `.dev.vars` の使い分け + `wrangler secret put` |

## 3. 開発フェーズ

各フェーズの実装タスクは **Sonnet サブエージェント**に分割発注する(オーケストレーションは本セッションが担当)。タスクは「仕様 → 実装 → テスト → レビュー」の粒度で切る。

### Phase 0: 技術検証(ゲート)

**目的**: jpoke × Cloudflare Python Workers の成立可否を実証し、§2.2 の方式を確定する。

| タスク | 内容 | 完了条件 |
|---|---|---|
| 0-1 | Python Workers 最小 PoC: jpoke を vendoring して `Battle.calc_damages()` を呼ぶ Worker を `wrangler dev` で起動 | ローカルで計算結果が jpoke ネイティブ実行と一致 |
| 0-2 | 実デプロイ検証: バンドルサイズ・コールドスタート・CPU 時間を計測 | サイズ制限内、p95 < 1.5s |
| 0-3 | (0-1/0-2 失敗時のみ) 案B ブラウザ Pyodide の PoC | 初回ロード時間の許容可否判断 |

**ゲート判定**: 結果を見て実行方式を確定してから Phase 1 に進む。

### Phase 1: 基盤構築

**目的**: 「共通データベースを育てる入口」の土台。以後の全機能がここに載る。

- 1-1 リポジトリ雛形: Astro + `@astrojs/cloudflare` + `wrangler.jsonc` + TypeScript strict + `worker.ts` 合成エントリ
- 1-2 Supabase ローカル開発環境: Supabase CLI (`supabase start`) + `migrations/` + 自前ランナー + `.env` 系ファイル整備
- 1-3 初期スキーマ: `events` / 型付きテーブル / `suggestions` + **全テーブル RLS**(§2.4)
- 1-4 イベントロギング API: `POST /api/events`(匿名 session_hash 発行・検証、レートリミット)
- 1-5 マスタデータ生成パイプライン: jpoke → フロント用軽量 JSON のビルドスクリプト
- 1-6 CI: GitHub Actions(`npm test` + postgres:16 マイグレーション検証)、Cloudflare Git 連携設定
- 1-7 計算サービス Worker の本組み込み(Phase 0 の PoC を製品化、Service Binding 接続)

### Phase 2: ダメージ計算機(最初の公開機能)

- 2-1 計算 API: `POST /api/calc/damage`(メインアプリ → 計算サービス → 結果 + 乱数16段階・致死率)
- 2-2 計算 UI: 攻撃側/防御側のポケモン・技・努力値(0〜32)・性格・持ち物・特性・テラス・フィールド状態入力、オートコンプリート
- 2-3 計算ログ蓄積: 計算実行ごとに `damage_calcs` + `events` へ匿名記録
- 2-4 「よく比較される育成 / 人気の仮想敵」表示枠: 当面は `suggestions` テーブル読み出しの空枠(Phase 5 で中身が入る)
- 2-5 E2E 検証: jpoke のテストケースから代表パターンを抽出し API 経由で回帰テスト

### Phase 3: 育成ビルダー

- 3-1 育成 CRUD API + 共有スラッグ(`/builds/[slug]` で閲覧可能な公開 URL)
- 3-2 ビルダー UI: 努力値配分・実数値表示(jpoke の `stats` 計算と一致させる)・技/持ち物/特性選択
- 3-3 ダメージ計算機との連携(作った育成をワンタップで計算機に読み込み)
- 3-4 保存・作成イベントの蓄積

### Phase 4: 検索

- 4-1 検索 API: ポケモン・技・特性・持ち物の横断検索(マスタ JSON ベース + Postgres 全文検索は必要になったら)
- 4-2 検索 UI + 詳細ページ(`/pokemon/[name]` など)
- 4-3 検索ログ蓄積 + 「最近注目されているもの」表示枠

### Phase 5: データ活用(集合知の還元)

- 5-1 集計 cron ジョブ: 1つの Cron Trigger + 分スロット分岐で「人気の育成 / よく比較される組み合わせ / 努力値傾向 / 注目検索」を日次集計し `suggestions` へ書き込み
- 5-2 各画面のサジェスト枠に集計結果を接続(Phase 2-4 で用意した空枠を埋める)
- 5-3 プライバシーレビュー: k-匿名性(表示は n 件以上集まったものだけ)の確認
- 5-4 評価スクリプト(`scripts/eval/`): 集計品質・subrequest 消費量の計測(poke-research の eval パターン)

将来(構想の「メタゲーム分析」「AI育成支援」等)は Phase 5 のデータが貯まってから別プランで扱う。

## 4. リスクと対応

| リスク | 影響 | 対応 |
|---|---|---|
| Python Workers で jpoke が動かない/遅い | アーキテクチャ根幹 | Phase 0 をゲートにし、案B(ブラウザ Pyodide)→案C(TS移植)のフォールバックを用意 |
| Cron Trigger アカウント上限(残り2枠) | 集計ジョブが登録できない | 1 Cron + 分スロット分岐に最初から集約(残り枠内で対処する方針で決定済み) |
| Worker subrequest 上限(50/呼び出し) | 集計ジョブの失敗 | poke-research の教訓どおりバッチ分割・`eval-subrequests` 相当の計測を Phase 5 に組み込み |
| 匿名データのプライバシー懸念 | 信頼毀損 | IP/UA 非保存・日替わり session_hash・k-匿名性表示を設計段階から適用 |
| jpoke 側の仕様変更(努力値形式など) | 計算結果の不整合 | jpoke をバージョン固定で vendoring し、更新は回帰テスト付きで取り込む |
| RLS 後付けの手戻り | セキュリティ指摘対応 | Phase 1 の初期マイグレーションから全テーブル RLS 有効 |

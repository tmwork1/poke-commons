作成日: 2026-07-23
前提資料: [初期構想.md](./初期構想.md)

## 1. ゴールとスコープ

初期構想の3機能 + データ蓄積基盤を、以下の順で段階的にリリースする。

1. **ダメージ計算機**(+ 計算ログの匿名蓄積)
2. **育成ビルダー**(保存・共有 + 蓄積)
3. **検索**(+ 検索ログ蓄積)
4. **データ活用**(蓄積データからのサジェスト表示)

対象ゲームは jpoke の対応範囲に合わせ **ポケモンチャンピオンズのシングルバトル**(努力値は 0〜32 のチャンピオンズ形式)。

「Data First」の思想に基づき、**イベントロギング基盤を Phase 1 で最初に作り**、全機能はその上に載せる。

## 2. アーキテクチャ

### 2.1 全体構成

```
[ブラウザ]
   │  Astro SSR ページ + フォーム/検索 UI
   │  + Pyodide 上の jpoke(クライアントサイド計算エンジン、静的アセット配信)
   ▼
[Cloudflare Workers: メインアプリ (TypeScript / Astro)]
   │  ├─ ファイルベース API (src/pages/api/**)
   │  ├─ middleware.ts (認証・レートリミット・CSRF)
   │  └─ worker.ts (fetch + scheduled 合成エントリ)
   │
   └──▶ [Supabase (Postgres)]
           マスタ参照キャッシュ・イベントログ(計算条件が中心)・育成データ・集計テーブル

[GitHub Actions (無料枠、定期実行)]
   │  ネイティブ Python + jpoke で damage_calcs のユニーク組み合わせを再計算・検証
   ▼
   Supabase へ検証済み結果を書き込み(suggestions)
```

- **メインアプリ**: poke-research と同じ Astro + `@astrojs/cloudflare` 構成。単一 npm パッケージ(モノレポにしない)。API/SSR のみを担い、ダメージ計算そのものは行わない。
- **計算エンジン**: jpoke をブラウザ内 **Pyodide** で実行(§2.2)。専用 Worker・Service Binding は持たない。
- **結果検証バッチ**: クライアント計算結果は未検証のまま蓄積し、信頼性が必要なサジェスト機能(§2.6)のみ GitHub Actions のネイティブ Python 実行でユニーク組み合わせを再計算して裏取りする。
- **DB**: Supabase (Postgres)。ローカル開発は Supabase CLI (`supabase start`, Docker) を使用(初期構想の指定どおり。poke-research には無い部分なので新規セットアップ)。

### 2.2 jpoke 実行方式 — 決定済み(2026-07-23)、Phase 0 は検証のみ

jpoke はランタイム依存ゼロ・C拡張なしの純 Python で、`numpy`/`pandas` 等の重量ライブラリに依存しないため Pyodide との相性が良い。Cloudflare Workers Free プランは **CPU時間 10ms/呼び出し(HTTP・Cron Trigger とも同一)、Worker バンドルサイズ 3MiB 上限** という制約があり、jpoke のマスタデータ単体で約3MBに達するため案Aは無料プランでは事実上成立しない([Cloudflare Workers Limits](https://developers.cloudflare.com/workers/platform/limits/))。この検証結果を踏まえ、**案Bをデフォルト方式として採用**する。

| 案 | 内容 | 懸念 |
|---|---|---|
| **B (採用)** | ブラウザ内 Pyodide で jpoke をクライアント実行 | 初回ロードサイズ(Pyodide コア約7MB + jpoke データ約3MB ≒ 10〜13MB想定)、低スペック端末での実行速度。ただしブラウザキャッシュ/Service Worker で初回以降は解消、サーバ負荷ゼロ・オフライン計算可 |
| A (非採用) | Python Workers で jpoke を動かす専用 Worker | Free プランの CPU 10ms/バンドル3MiB制限にほぼ確実に抵触。成立させるには Workers Paid(月$5〜)が前提になる |
| C (最終手段) | ダメージ計算コア部分のみ TypeScript へ移植 | 移植工数大・jpoke の 6174 テストとの等価性維持が困難 |

**Phase 0 の役割**: 案Bの実現可否(ロードサイズ・初期化時間・計算結果の一致)を検証する。想定を超えて破綻した場合のみ、Paid プラン前提で案A、それも難しければ案Cを検討する。

### 2.3 マスターデータの流れ

jpoke の `src/jpoke/data/`(pokedex.json / moves.json / learnsets.json / CSV)を**単一の信頼できる情報源**とし、ビルド時スクリプトで以下2種類の静的アセットを生成する。DB にはマスタを複製せず、**イベントログには名前(jpoke の日本語キー)をそのまま記録**する。

1. **オートコンプリート用軽量 JSON**(ポケモン名・技名・特性・アイテム): フォーム入力補助向け
2. **Pyodide 実行用フルパッケージ**(jpoke 本体 + データ一式): ブラウザ内計算エンジン向け。Cloudflare の静的アセット配信はリクエスト数/CPU時間の制約を受けないため、Free プランのままサイズ・帯域の心配なく配信できる

### 2.6 計算結果の信頼性方針 — 決定済み(2026-07-23)

計算をクライアント(Pyodide)で行うため、サーバは計算結果を検証できない。悪意あるクライアントが捏造した結果を送信すると、集合知の集計データ(サジェスト)が汚染されるリスクがある。これに対応するため、初期構想の機能を「計算条件(入力)だけで作れるか」で仕分けて扱いを分ける。

| 分類 | 該当機能 | 扱い |
|---|---|---|
| **入力ベース(大半)** | よく比較される育成/ポケモン、人気の仮想敵、人気の育成、採用率、技構成の傾向、努力値の傾向、最近注目されているもの | `POST /api/events` は**計算条件のみ**を正データとして受け付け、集計もそこからのみ行う。結果の真正性に依存しないため無検証で提供可能 |
| **結果依存(少数)** | 勝率の高い調整、おすすめの耐久ライン | クライアント計算結果(`client_result`)は UI 表示用の未検証値として保存するのみで、集計には使わない。Phase 5 で GitHub Actions の**ネイティブ Python + jpoke**(Pyodide 経由ではないため CPU時間制限なし)による**ユニークな組み合わせ単位のバッチ再計算**を導入し、検証済みの値のみ `suggestions` へ書き込んでから提供する |

Cloudflare Workers 上での検証(Cron Trigger 含む)は Free プランの CPU 10ms/呼び出し制限に阻まれるため選ばない。GitHub Actions(private リポジトリでも月2,000分無料)ならネイティブ実行のため制限を受けない。

### 2.4 データモデル(初期案 — Phase 1 で確定)

```sql
-- 匿名イベントログ (append-only、全機能共通の蓄積基盤)
events (
  id bigint generated always as identity,
  event_type text,          -- 'damage_calc' | 'build_save' | 'search' | 'compare' | ...
  payload jsonb,            -- イベント固有データ (計算条件、検索語 など)
  session_hash text,        -- 匿名セッション識別子 (日次ローテーションのハッシュ、個人特定不可)
  created_at timestamptz default now()
)

-- 型付きテーブル (集計しやすい主要イベントは正規化して二重記録)
damage_calcs ( attacker_name, defender_name, move_name, attacker_build jsonb,
               defender_build jsonb, field jsonb,
               client_result jsonb,      -- クライアント計算結果。UI表示用の未検証値、集計には使わない(§2.6)
               verified_result jsonb,    -- GitHub Actions バッチが埋める検証済み結果(Phase 5)
               session_hash, created_at )

builds ( id uuid, pokemon_name, nature, ability_name, item_name, tera_type,
         evs jsonb, ivs jsonb, move_names text[], share_slug text unique,
         is_public boolean, session_hash, created_at )

searches ( query text, category text, hit_count int, session_hash, created_at )

-- 集計結果 (cron ジョブが書き込み、閲覧 API は読むだけ)
suggestions ( kind text, subject_key text, payload jsonb, computed_at )
  -- kind 例: 'popular_build' | 'common_opponent' | 'ev_trend' | 'trending_search'
```

- **RLS を Phase 1 の最初から全テーブルに適用**(poke-research では後付け対応 `027_enable_rls.sql` になった教訓)。閲覧系は `SELECT` のみ public、書き込みは service_role 経由の API のみ。
- 匿名化ポリシー: IP・UAは保存しない。`session_hash` は日替わりソルト付きハッシュで長期追跡不可にする。

### 2.5 poke-research から流用するパターン

| 項目 | 流用内容 |
|---|---|
| マイグレーション | `migrations/NNN_*.sql` 連番 + `scripts/db/run-migrations.mjs`(`pg` 直結の自前ランナー、適用済み管理テーブル) |
| API 設計 | Astro ファイルベースエンドポイント + `api/_shared.ts` 共通ヘルパー。Hono 等は導入しない |
| 認証 | 管理系 API のみ Basic 認証(`middleware.ts`)。一般ユーザーはログイン不要(匿名) |
| Cron | `worker.ts` に fetch + scheduled 合成エントリ。**Cron Trigger の上限は Free プランでアカウントごと5件**([公式 Limits](https://developers.cloudflare.com/workers/platform/limits/))で、poke-research が3件使用済み。**残り枠内で対処する方針(決定済み 2026-07-23)**: poke-commons は1件に集約し分スロットでジョブ分岐(poke-research の `DAILY_SLOT_JOBS` パターン) |
| テスト | `node --test`(外部フレームワーク無し)+ CI で postgres:16 コンテナへのマイグレーション DDL 検証 |
| デプロイ | Cloudflare Git 連携で main ブランチ自動デプロイ。本番マイグレーションのみ `npm run release` で手動先行適用 |
| 環境変数 | `.env.example` / `.env` / `.env.production` / `.dev.vars` の使い分け + `wrangler secret put` |

- [x] **0-3** (0-1/0-2 失敗時のみ)案A/案Cの再検討
  - 結果: 致命的な不成立ではないため不要と判断。案Bを維持し、緩和策をPhase 1のタスクとして追加

**ゲート判定: 案B(ブラウザ内 Pyodide でのクライアント実行)を採用として確定。** モバイル低速回線での初回ロードが遅い課題は、Phase 1-7 に以下の緩和策を組み込むことで対応する(致命的な破綻ではないため案A/Cへの切替は不要):

## 4. リスクと対応

| リスク | 影響 | 対応 |
|---|---|---|
| ブラウザ Pyodide の初回ロードが重い/低スペック端末で遅い | UX 低下・離脱率上昇 | Phase 0 でロードサイズ・体感速度を実測してゲート判定。ローディング UI・バックグラウンドプリロード・Service Worker / IndexedDB キャッシュで初回以降を高速化。想定を超えて破綻した場合のみ Paid プラン前提の案A、または案C(TS移植)を検討 |
| クライアント計算結果の捏造・改ざん | 集計データ(サジェスト)の汚染 | 入力ベースの機能(大半)は結果に依存しない設計にし、結果依存の機能(勝率の高い調整・耐久ライン)は GitHub Actions によるユニーク組み合わせ単位のサーバ再計算で検証してから提供(§2.6) |
| Cron Trigger アカウント上限(残り2枠) | 集計ジョブが登録できない | 1 Cron + 分スロット分岐に最初から集約(残り枠内で対処する方針で決定済み)。Cloudflare Free プランは HTTP・Cron とも CPU 10ms/呼び出しのため、重い再計算はここに乗せず GitHub Actions へ分離(§2.6) |
| Worker subrequest 上限(50/呼び出し) | 集計ジョブの失敗 | poke-research の教訓どおりバッチ分割・`eval-subrequests` 相当の計測を Phase 5 に組み込み |
| 匿名データのプライバシー懸念 | 信頼毀損 | IP/UA 非保存・日替わり session_hash・k-匿名性表示を設計段階から適用 |
| jpoke 側の仕様変更(努力値形式など) | 計算結果の不整合 | jpoke をバージョン固定で vendoring し、更新は回帰テスト付きで取り込む |
| RLS 後付けの手戻り | セキュリティ指摘対応 | Phase 1 の初期マイグレーションから全テーブル RLS 有効 |
